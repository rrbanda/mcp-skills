# On-Prem AI Coding Agent on OpenShift

A reference architecture for running [goose](https://block.github.io/goose/) -- an open-source AI coding agent by Block -- fully on-premises on OpenShift. Goose writes code, debugs, generates MCP tools, deploys to the cluster, and fixes its own bugs -- all powered by an on-prem LLM served via Llama Stack on OpenShift AI.

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  OpenShift Dev Spaces (VS Code in Browser)                  │
│  ┌───────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │  Goose Agent   │  │  LLM Proxy   │  │  Skills/Recipes  │  │
│  │  (CLI + ext)   │  │  :9090       │  │  (YAML files)    │  │
│  └───────┬───────┘  └──────┬───────┘  └──────────────────┘  │
│          │                 │                                  │
└──────────┼─────────────────┼────────────────────────────────┘
           │                 │
           │                 ▼
           │  ┌──────────────────────────────────┐
           │  │  OpenShift AI (Llama Stack)       │
           │  │  vLLM → gpt-oss-120b (on-prem)   │
           │  │  Gemini (cloud, optional)         │
           │  └──────────────────────────────────┘
           ▼
  ┌────────────────────────────┐
  │  MCP Servers (tools)       │
  │  • DevSpaces MCP           │  ← generated by goose
  │  • kubernetes-mcp-server   │  ← pre-deployed (optional)
  │  • Your MCP Server         │  ← add your own
  └────────────┬───────────────┘
               ▼
  ┌────────────────────────────┐
  │  OpenShift Cluster APIs    │
  │  DevWorkspace, K8s, etc.   │
  └────────────────────────────┘
```

## Prerequisites

| Component | Required | Notes |
|-----------|----------|-------|
| OpenShift 4.x cluster | Yes | With admin access |
| OpenShift Dev Spaces operator | Yes | Install from OperatorHub |
| OpenShift AI (RHOAI) | Yes | For serving the LLM |
| Llama Stack | Yes | OpenAI-compatible gateway on RHOAI |
| vLLM model serving | Yes | e.g. `gpt-oss-120b` with `max_model_len: 65536` |

### Find your Llama Stack URL and model name

Before starting, you need two pieces of information from your OpenShift AI setup:

**1. Llama Stack endpoint URL**

```bash
# Find the Llama Stack route (adjust namespace to where Llama Stack is deployed)
oc get route -n llamastack -o jsonpath='{.items[*].spec.host}'
```

Your URL will look like: `https://llamastack-llamastack.apps.<cluster-domain>`

**2. Available model names**

```bash
curl -s https://<your-llamastack-host>/v1/models | python3 -m json.tool
```

Look for entries with `"model_type": "llm"`. The model `id` field (e.g. `vllm-inference/gpt-oss-120b`) is what you'll use for `GOOSE_MODEL`.

## Quick Start (no fork needed)

This gets you running in under 10 minutes using the pre-built workspace image.

### Step 1: Open Dev Spaces and create the workspace

1. Log in to the **OpenShift Console**
2. Click the **grid icon** (top-right) and select **Red Hat OpenShift Dev Spaces**
   - Or navigate directly to `https://devspaces.apps.<your-cluster-domain>/`
3. Click **Create Workspace**
4. Paste the Git repo URL:
   ```
   https://github.com/rrbanda/mcp-skills.git
   ```
5. Click **Create & Open**

Dev Spaces reads [devfile.yaml](devfile.yaml), pulls the pre-built image (`quay.io/rbrhssa/mcp-skills:latest`), and opens VS Code in your browser. Wait for the workspace to finish starting (1-2 minutes).

### Step 2: Configure for your cluster

Open a terminal in Dev Spaces (**Terminal > New Terminal**) and configure two things:

**a) Set your Llama Stack URL**

```bash
# Replace with your actual Llama Stack URL from the prerequisites step
export LLAMASTACK_URL=https://llamastack-llamastack.apps.<your-cluster-domain>

# Verify it's reachable
curl -s $LLAMASTACK_URL/v1/models | python3 -c "import sys,json; [print(m['id']) for m in json.load(sys.stdin)['data'] if m.get('custom_metadata',{}).get('model_type')=='llm']"
```

**b) Set the model name** (if different from the default)

```bash
# Only needed if your model is NOT named vllm-inference/gpt-oss-120b
export GOOSE_MODEL=<your-model-id>
```

### Step 3: Verify environment variables

```bash
echo "OPENAI_HOST=$OPENAI_HOST"
echo "OPENAI_BASE_PATH=$OPENAI_BASE_PATH"
echo "GOOSE_PROVIDER=$GOOSE_PROVIDER"
echo "LLAMASTACK_URL=$LLAMASTACK_URL"
```

All four should have values. **If `OPENAI_HOST` or `OPENAI_BASE_PATH` are empty**, the devfile env vars weren't applied -- set them all:

```bash
export OPENAI_HOST=http://localhost:9090
export OPENAI_BASE_PATH=v1/responses
export GOOSE_PROVIDER=openai
export GOOSE_MODEL=vllm-inference/gpt-oss-120b
export OPENAI_API_KEY=dummy
export XDG_STATE_HOME=/tmp/.local/state
```

> **Tip:** Add all exports to `~/.bashrc` so they persist across terminal restarts:
> ```bash
> cat >> ~/.bashrc << 'EOF'
> export LLAMASTACK_URL=https://llamastack-llamastack.apps.<your-cluster-domain>
> export OPENAI_HOST=http://localhost:9090
> export OPENAI_BASE_PATH=v1/responses
> export GOOSE_PROVIDER=openai
> export GOOSE_MODEL=vllm-inference/gpt-oss-120b
> export OPENAI_API_KEY=dummy
> export XDG_STATE_HOME=/tmp/.local/state
> EOF
> ```

### Step 4: Verify the LLM proxy is running

The [llm_proxy.py](llm_proxy.py) proxy starts automatically when the workspace launches (via the devfile `postStart` event). Verify it's running and can reach Llama Stack:

```bash
curl -s http://localhost:9090/v1/models | python3 -c "import sys,json; print('OK -', len(json.load(sys.stdin)['data']), 'models available')"
```

You should see something like `OK - 3 models available`.

If it's not running (connection refused), start it manually:

```bash
nohup python3 llm_proxy.py > /tmp/llm-proxy.log 2>&1 &
```

Check logs anytime with `cat /tmp/llm-proxy.log`.

### Step 5: Install the Goose VS Code extension (optional)

If you want the goose sidebar UI in VS Code:

```bash
/checode/checode-linux-libc/bin/remote-cli/code \
  --install-extension /opt/extensions/vscode-goose.vsix
```

Or use **Terminal > Run Task > install-goose-extension**. The CLI works without this.

### Step 6: Verify goose is working

Test basic LLM connectivity:

```bash
goose run --no-session --text "Hello, what model are you?"
```

Test tool calling (goose should invoke the `shell` tool):

```bash
goose run --no-session --text "Run ls and tell me what files are here"
```

If both work, your setup is complete.

## Demo: Generate and Deploy an MCP Server

This demo shows goose generating an MCP server from a REST API, then deploying it to OpenShift -- entirely driven by skills, no manual coding.

### Demo Step 1: Generate the MCP server

```bash
goose run \
  --recipe skills/create-fastmcp-server.yaml \
  --params output_filename=devspaces_mcp.py \
  --params "server_name=DevSpaces MCP Server" \
  --no-session
```

Goose reads [.goosehints](.goosehints) for API context, generates `devspaces_mcp.py` with MCP tools for workspace management (list, get, create, start, stop, delete), creates `requirements.txt`, validates imports, and smoke tests against the live cluster.

### Demo Step 2: Test the generated server locally

```bash
goose run --no-session \
  --text "Test the list_workspaces tool in devspaces_mcp.py by running it against the admin-devspaces namespace. If there are bugs, fix them."
```

Goose will run the code, find bugs (if any), and fix them autonomously.

### Demo Step 3: Register and use the MCP server

Add the generated server as a goose extension:

```bash
goose configure
# Choose: Add Extension > stdio
# Command: python3
# Args: devspaces_mcp.py
# Name: devspaces-mcp
```

Then use it:

```bash
goose run --no-session --text "List all Dev Spaces workspaces in admin-devspaces"
```

### Demo Step 4: Deploy the MCP server to OpenShift

**a) Grant build permissions** (one-time setup, run from the Dev Spaces terminal):

```bash
SA=$(oc get pod -l 'controller.devfile.io/devworkspace_name' \
  -n admin-devspaces -o jsonpath='{.items[0].spec.serviceAccountName}')

oc adm policy add-role-to-user edit \
  system:serviceaccount:admin-devspaces:$SA -n admin-devspaces
```

**b) Run the deploy skill** -- goose handles everything:

```bash
goose run \
  --recipe skills/deploy-fastmcp-server.yaml \
  --params server_file=devspaces_mcp.py \
  --params namespace=admin-devspaces \
  --params image_name=devspaces-mcp \
  --no-session
```

Goose will:
1. Read the server code and requirements
2. Create a deploy directory with HTTP transport enabled
3. Generate a Dockerfile
4. Build the image using OpenShift's native build system (`oc new-build` / `oc start-build`)
5. Create Deployment, Service, and Route via `oc apply`
6. Verify the pod is Running
7. Report the URL: `https://devspaces-mcp-<namespace>.apps.<cluster>/mcp`

### Demo Step 5: Use the deployed MCP server

Register the deployed server as a goose extension:

```bash
goose configure
# Choose: Add Extension > Remote (streamable-http)
# URL: https://devspaces-mcp-admin-devspaces.apps.<your-cluster-domain>/mcp
# Name: devspaces-mcp-remote
```

Then use it:

```bash
goose run --no-session --text "List all workspaces in admin-devspaces namespace"
```

## Using a cloud model via Llama Stack

If the on-prem model's context window is too small for complex tasks, Llama Stack can proxy to cloud models like Gemini. Override the model at runtime:

```bash
GOOSE_MODEL="gemini/models/gemini-2.5-flash" goose run --no-session --text "your task"
```

The request still goes through the local proxy to Llama Stack, which routes to Gemini. No API keys are needed in the workspace -- Llama Stack handles auth server-side.

## Custom Setup (optional)

If you want to use your own container image or fork:

### Build and push a custom workspace image

```bash
# Clone and modify
git clone https://github.com/rrbanda/mcp-skills.git
cd mcp-skills

# Edit devfile.yaml: change the image and LLAMASTACK_URL
# Edit llm_proxy.py is NOT needed -- it reads LLAMASTACK_URL from the env var

# Build for linux/amd64 (OpenShift clusters are x86_64)
podman build --platform linux/amd64 -t quay.io/<your-org>/mcp-skills:latest .
podman push quay.io/<your-org>/mcp-skills:latest
```

Update [devfile.yaml](devfile.yaml) to reference your image and Llama Stack URL:

```yaml
components:
  - name: tools
    container:
      image: quay.io/<your-org>/mcp-skills:latest
      env:
        - name: LLAMASTACK_URL
          value: https://<your-llamastack-host>
        - name: GOOSE_MODEL
          value: <your-model-id>
```

Then push to your fork and create the workspace from your fork URL.

### Adding Your Own Skills

Skills are YAML recipe files that guide goose through multi-step tasks. See the [goose recipe docs](https://block.github.io/goose/docs/guides/recipes/).

Key patterns from this repo:
- **Skill** = reusable workflow (e.g., "how to build an MCP server")
- **`.goosehints`** = project-specific context (e.g., API endpoints, auth patterns)
- **Explicit tool lists** in skill `instructions` prevent the LLM from hallucinating tool names
- **Step-by-step prompts** ensure on-prem models follow the full workflow

## Troubleshooting

### Env vars not set after workspace creation

If `OPENAI_HOST` or `OPENAI_BASE_PATH` are empty, the devfile env vars weren't applied (happens when the workspace was created before the devfile was updated). Export them manually (see Step 3) or delete and recreate the workspace.

### goose sends requests to api.openai.com instead of Llama Stack

`OPENAI_HOST` is not set -- goose defaults to `https://api.openai.com`. Fix:

```bash
export OPENAI_HOST=http://localhost:9090
```

### goose says "Authentication error: 401"

Same root cause -- requests going to OpenAI.com with the dummy key. Verify `OPENAI_HOST` points to `http://localhost:9090`.

### "Permission denied" writing goose logs

Set `XDG_STATE_HOME=/tmp/.local/state` (already in devfile).

### Streaming parse errors from the LLM

The LLM proxy handles known issues. If new errors appear, check proxy logs and look at `fix_sse_event()` in [llm_proxy.py](llm_proxy.py).

### goose hallucinates tool names (open_file, apply_patch)

On-prem models do this when tool lists aren't explicit. The skills in this repo include an "Available Tools" section. When running without a skill, use `--system`:

```bash
goose run --no-session \
  --system "You have ONLY these tools: write, edit, shell, tree. NEVER call open_file or apply_patch." \
  --text "your task here"
```

### oc start-build fails with "Forbidden"

The workspace service account needs build permissions:

```bash
SA=$(oc get pod -l 'controller.devfile.io/devworkspace_name' \
  -n <namespace> -o jsonpath='{.items[0].spec.serviceAccountName}')
oc adm policy add-role-to-user edit \
  system:serviceaccount:<namespace>:$SA -n <namespace>
```

### Model context too small for complex tasks

The on-prem model needs `max_model_len: 65536`. For tasks needing larger context, switch to a cloud model via Llama Stack:

```bash
GOOSE_MODEL="gemini/models/gemini-2.5-flash" goose run ...
```

### podman build fails inside Dev Spaces

Container builds inside Dev Spaces are restricted (no `clone` syscall). The deploy skill uses OpenShift's native build system (`oc new-build --binary`) instead.
