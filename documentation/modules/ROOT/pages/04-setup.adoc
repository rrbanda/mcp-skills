= Setting Up the Environment
include::_attributes.adoc[]

== Overview

We'll deploy the full Agent Sandbox environment using a Goose AI recipe. This demonstrates production-grade infrastructure-as-code via AI agent.

== Option A: Using the Goose Recipe (Recommended)

Run the deploy-agent-sandbox recipe:

[.console-input]
[source,bash]
----
goose run --recipe skills/deploy-agent-sandbox.yaml
----

TIP: The recipe defaults to namespace `{sandbox-namespace}` and 3 warm pool replicas. Override with `--params`: `goose run --recipe skills/deploy-agent-sandbox.yaml --params namespace=my-ns --params warmpool_replicas=5`

The recipe performs the following steps:

* Installs the agent-sandbox operator (if not already present)
* Creates the target namespace and grants OpenShift SCC permissions
* Deploys a SandboxTemplate with Kata VM isolation
* Applies a NetworkPolicy for deny-all egress (DNS only)
* Creates a WarmPool with 3 replicas for instant sandbox allocation
* Verifies CRDs, controller, RuntimeClass, and warm pool pods

== Option B: Manual Setup

For those who prefer scripts, run:

[.console-input]
[source,bash]
----
./demo/setup.sh
----

Or perform the steps manually:

. Install agent-sandbox operator and extensions:
+
[.console-input]
[source,bash]
----
oc apply -f https://github.com/kubernetes-sigs/agent-sandbox/releases/download/{agent-sandbox-version}/manifest.yaml
oc apply -f https://github.com/kubernetes-sigs/agent-sandbox/releases/download/{agent-sandbox-version}/extensions.yaml
----

. Create the namespace:
+
[.console-input]
[source,bash]
----
oc create namespace {sandbox-namespace}
----

. **(OpenShift only)** Grant SCC permissions so the controller can create pods with the specified `runAsUser`:
+
[.console-input]
[source,bash]
----
oc adm policy add-scc-to-user anyuid -z agent-sandbox-controller -n agent-sandbox-system
oc adm policy add-scc-to-user anyuid -z default -n {sandbox-namespace}
----
+
NOTE: On vanilla Kubernetes this step is not needed. OpenShift's `restricted-v2` SCC rejects pods whose `runAsUser` falls outside the namespace's auto-assigned UID range.

. Apply the SandboxTemplate:
+
[.console-input]
[source,bash]
----
oc apply -f demo/sandbox-template.yaml
----

. Apply the WarmPool:
+
[.console-input]
[source,bash]
----
oc apply -f demo/sandbox-warmpool.yaml
----

. Apply the NetworkPolicy for egress isolation:
+
[.console-input]
[source,bash]
----
oc apply -f demo/sandbox-networkpolicy.yaml
----
+
NOTE: The SandboxTemplate defines a `networkPolicy` section, but the agent-sandbox controller (v0.2.1) does not auto-create Kubernetes NetworkPolicy resources from it. This manual step ensures the deny-all egress (DNS only) policy is enforced.

== Understanding the SandboxTemplate

The SandboxTemplate defines the security posture for each sandbox pod:

----

=== Core Settings (all environments)

[cols="2,3"]
|===
| Setting | Purpose

| `automountServiceAccountToken: false`
| Prevents Kubernetes API access — the sandbox cannot read secrets or call the cluster API.

| `runAsNonRoot: true` / `runAsUser: 10001`
| Runs as non-root user. On OpenShift, granting `anyuid` SCC is required because the hardcoded UID 10001 falls outside the namespace's auto-assigned range.

| `resources.limits`
| CPU and memory caps per sandbox (2 CPU, 4Gi memory in this template).

| `networkPolicy`
| Deny-all egress except DNS (ports 53). Ingress only from the workspace namespace. Prevents data exfiltration and lateral movement.
|===

NOTE: As of agent-sandbox v0.2.1, the controller does **not** auto-create Kubernetes NetworkPolicy resources from the template's `networkPolicy` section. You must apply `demo/sandbox-networkpolicy.yaml` separately (step 6 above). This may be fixed in a future release.

IMPORTANT: The `demo/sandbox-template.yaml` file ships configured for **AWS peer-pods**. If you are running on bare metal, edit it to change `runtimeClassName` to `kata` and remove the `machine_type` annotation.
